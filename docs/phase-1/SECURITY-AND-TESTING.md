# Phase 1 security and testing

## Tenant isolation

RLS is enabled for every tenant-owned table. Helper functions check active membership and explicit roles. Internal staging, active-transaction, issue, and audit inserts occur only through validated security-definer RPCs with fixed `search_path`.

## Storage

Source files live in a private R2 bucket. Object keys are generated by the worker from verified organization and import records plus the source SHA-256. This makes source objects content-addressed and prevents a different retry from overwriting prior evidence. The browser cannot choose another tenant's path.

## Sensitive logging

Operational logs may contain IDs, stages, counts, durations, parser versions, and error codes. They must not contain raw CSV rows, descriptions, account numbers, signed URLs, or file contents.

## Prompt injection

CSV fields and transaction descriptions are never treated as instructions. Phase 1 performs no model call. Future AI phases must pass imported text only as quoted structured data.

## Test coverage

The suite verifies:

- allowed and forbidden state transitions;
- integer minor-unit money parsing;
- Indian grouping, parentheses, and DR/CR suffixes;
- encoding, delimiters, header detection, quotes, and ambiguous dates;
- strong versus fuzzy duplicate treatment;
- invalid dates and visible rejection;
- inert prompt-injection descriptions;
- R2 rollback after metadata failure;
- exact-file duplication rejection;
- content-addressed object keys and upload-state protection;
- public health endpoint behavior;
- migration-level RLS and atomic RPC guardrails.

## Required pre-production checks

- Apply migration to an isolated Supabase project.
- Execute two-user cross-tenant tests against real RLS.
- Inject failures before and during commit.
- Verify R2 source objects are private.
- Confirm no raw financial data appears in Worker logs.
- Load-test representative 1k, 10k, and 50k row files.
